1. 架构演进

  • 单一应用架构（All in One）
    • 网站用户较少，只需要一个应用，将所有功能部署在一起，以减少部署节点和成本。
    • 关键是ORM框架。
  • 垂直应用架构（集群）
    • 当访问增大时，单一应用增加机器带来的加速越来越小，将应用拆成互不相干的几个应用，以提升效率。
    • MVC分层架构是关键。
  • 分布式服务架构（服务拆分）
    • 当垂直应用越来越多，应用之间交互不可避免，将核心业务抽取出来作为独立的服务，逐渐形成稳定的服务中心，使前端应用能更快速的响应多变的市场需求。
    • 服务复用和服务通讯（RPC）是关键。
  • 流动计算架构（服务治理）
    • 当服务越来越多，容量的评估，小服务资源的浪费等问题逐渐显现，此时需增加一个调度中心基于访问压力实时管理集群容量，提高集群利用率。
    • 提高机器利用率的资源调度和治理中心（SOA）是关键。

2. 架构

2.1 角色

节点 角色说明
Provider 服务提供者（暴露服务的服务提供方）
Consumer 调用远程服务的服务消费方
Registry 服务注册与发现的注册中心
Monitor 统计服务调用次数和调用时间的监控中心
Container 服务运行容器

  1. Container负责启动，加载，运行服务提供者。
  2. Provider启动时向Registry注册自己提供的服务。
  3. Consumer启动时向Registry订阅自己需要的服务。
  4. Registry返回Provider地址列表给Consumer，如有变更，Registry将基于长连接推送变更数据给Consumer。
  5. Consumer从Provider地址列表中，基于软负载均衡算法，选一台Provider进行服务调用，如果调用失败，再选另外一台调用。
  6. Provider和Consumer在内存累计调用次数和调用时间，定时每分钟发送一次统计数据到Monitor。

2.2 连通性

  • 注册中心负责服务的注册和查找，相当于目录服务，服务提供者和服务消费者只在启动时与注册中心交互，注册中心不转发请求，压力较小。
  • 监控中心负责统计调用次数与调用时间，统计先在内存汇总，定时（每分钟）发送给监控中心，并以报表展示。
  • 服务提供者向注册中心注册其提供的服务，并汇报调用时间到监控中心，此时间不包含网络开销。
  • 服务消费者向注册中心获取服务提供者地址列表，根据负载方法直接调用提供者，并向监控中心汇报调用时间，此时间段包含网络开销。
  • 注册中心，服务提供者，服务消费者三者之间都是长连接，监控中心除外。
  • 注册中心通过长连接感知服务提供者的存在，如果服务提供者宕机，注册中心将立即推送事件通知消费者。
  • 注册中心和监控中心全部宕机，不影响已运行的提供者和消费者，消费者在本地缓存了提供者列表地址。
  • 注册中心和监控中心都是可选的，服务消费者可以直连服务提供者。

2.3 健壮性

  • 监控中心宕机不影响使用，只丢失部分采样数据。
  • 数据库宕机后，注册中心仍能通过缓存提供服务列表查询，但不能注册新服务。
  • 注册中心对等集群，任意一台宕机后，会自动切换到另一台。
  • 注册中心宕机后，服务消费者仍然可以通过本地缓存与服务提供者通讯。
  • 服务提供者是无状态的，任意一台宕机掉，不影响使用。
  • 服务提供者全部宕机，服务消费者应用将不能使用，并无限次重连等待服务提供者恢复。

2.4 伸缩性

  • 注册中心为对等集群，可动态增加部署实例，所有客户端将自动发现新的注册中心。
  • 服务提供者无状态，可动态增加机器部署实例，注册中心将推送新的服务提供者信息给消费者。

4. 原理

4.1 Java SPI

  • SPI全称Service Provider Interface，是一种服务发现。SPI的本质是将接口的实现类的全限定名配置在文件中，并由服务加载器读取配置文件，加载实现类。这样可以在运行时，动态为接口替换实现类。

  • Java SPI代码示例

    • 接口定义和实现类，接口添加@SPI注解

    • SPI配置文件（接口的全限定名作为文件名）

4.2 Dubbo SPI

  • dubbo重新实现了一套功能更强的SPI，封装在ExtensionLoader类中，通过ExtensionLoader可以加载指定的接口实现类。

  • 代码示例

    • 配置文件放在META-INF/dubbo路径下，名称为接口全路径名

4.4 自适应扩展

  • Adaptive注解

    • Adaptive可注解在类或方法上。当注解在类上时，不会生成代理类；在方法上时，会为该方法生成代理逻辑。
    • Adaptive注解在类上的情况很少，在Dubbo中，只有AdaptiveCompiler和AdaptiveExtensionFactory。

4.5 服务导出

4.5.1 简介

​ Dubbo服务导出过程始于spring容器发布刷新事件，Dubbo在接收到事件后，会立刻执行服务导出逻辑。整个导出逻辑分为三个部分：

  • 前置动作，检查参数，组装URL。
  • 导出服务，包含导出到本地（JVM）和导出到远程两个过程。
  • 向注册中心注册服务，用户服务发现。

4.5.2 前置动作

  • 检测配置

4.6 服务引用

​ 服务引用的时机有两个，第一个是在容器中调用ReferenceBean的afterPropertiesSet()方法，第二个是ReferenceBean对应的服务被注入到其他类中时引用。这两种引用服务的时机区别在于第一个是饿汉式，第二个懒汉式的。Dubbo默认是懒汉式的。如果需要饿汉式，可通过配置<dubbo:reference> 的init属性开启。

​ 分析默认的方式，当我们的服务被注入到其他类中时，Spring会第一时间调用getObject()方法，并由该方法执行服务引用的逻辑。

  • ReferenceBean#getObject()

  • DubboProtocol#refer()

  • RegistryProtocol#refer()

  • 创建代理–proxyFactory.getProxy(invoker)

4.7 服务调用过程

  • 消费者通过代理对象Proxy发起远程调用。
  • 通过网络客户端Client将编码后的请求发送给服务提供方的网络层Server。
  • Server接收请求后，对数据包解码。
  • 将解码后的请求发送至分发器Dispatcher。
  • 分发器将请求派发到指定线程池上，线程池调用服务。

4.7.1 服务调用方式

​ Dubbo支持同步和异步两种调用方式，异步还可以根据是否有返回值区分。无返回值，消费方只管调用，Dubbo会直接返回空的RpcResult。Dubbo默认同步调用。

  • Dubbo采用Javassist框架为服务接口生成动态代理类。

4.7.2 服务消费方发送请求

  • 发送请求

  • 请求编码

    • Dubbo数据包包含消息头和消息体，消息头用于存储一些元信息，比如魔数、数据包类型、消息体长度等。消息体用于存储具体的调用信息，比如方法名称、参数列表等。

    • 首先通过位运算将消息头写入header数组中，然后对request中的data字段执行序列化，序列化后的数据存入ChannelBuffer中。序列化操作执行完成之后，可得到序列化后的长度len，将len写到header指定位置处。

4.7.3 服务提供方接收请求

  • 请求解码

    • 通过检测消息头中的魔数是否与规定魔数相同，提前拦截非常规数据包（比如通过telnet命令行发生数据包）。
    • 对消息体长度，以及可读字节数进行检测，最后通过DubboCodec#decodeBody()进行后续解码。
    • decodeBody 对部分字段进行了解码，并将解码得到的字段封装到 Request 中 。
    • 调用 DecodeableRpcInvocation 的 decode 方法进行后续的解码工作。此工作完成后，可将调用方法名、attachment、以及调用参数解析出来 。

  • 调用服务

    • 线程派发模型

      • Dubbo将底层通信框架中接收请求的线程称为IO。如果一些事件处理逻辑很快执行完，比如在内存打个标记，此时直接在IO线程上执行该段逻辑。如果事件处理逻辑比较耗时，比如HTTP请求，此时将请求派发到线程池中执行。

      • Dispatcher是线程派发器 。Dispatcher的真实职责是创建具有线程派发能力的ChannelHandler，其本身不具备线程派发能力。Dubbo支持5种不同的线程派发策略：

        策略 用途
        all 所有消息都被派发到线程池，包括请求，响应，连接事件，断开事件等
        direct 所有线程都不派发到线程池，全部在IO线程上执行
        message 只有请求和响应消息派发到线程池，其余直接在IO上执行
        execution 只有请求消息，不含响应派发到线程池，其余直接在IO上执行
        connection 在IO线程上，将连接断开事件放入队列，有序逐个执行，其它消息派发到线程池

      • Dubbo默认all派发策略

      • AllChannelHandler

    • 调用服务（ChannelEventRunnable）

4.7.4 服务提供方返回调用结果

​ 服务提供方调用指定服务后，会将处理结果封装到Response，并将该对象进行编码再返回给服务消费方。

4.7.5 服务消费方接收调用结果

  • 响应数据解码（同request解码）

  • 向用户线程传递调用结果（通过调用编号获取自己的响应对象）
